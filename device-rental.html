Device Rental Service

Сервис аренды игровых девайсов для гостей клуба. Гость выбирает мышь, клавиатуру или гарнитуру на своём столе, инженер выдаёт через Telegram-бота, склад пересчитывается сам, а игровые сессии сверяются со SmartShell — чтобы устройство не «уехало» вместе с ушедшим гостем.

[ Задача ]

Что нужно было решить

В клубе есть парк сменных девайсов — мыши, клавиатуры, гарнитуры, кабели, повербанки. До сервиса их выдача держалась на памяти смены: кто взял, на какой ПК, вернул ли. Устройства терялись, а гость ждал, пока свободный инженер добежит до склада.

Нужна была система, где гость сам оформляет заявку со своего ПК, инженер выдаёт в пару нажатий прямо в Telegram, а остаток на складе всегда сходится — без ручного пересчёта и «минус одной мыши» в конце смены.

Отдельное требование — привязка к SmartShell: если игровая сессия гостя закончилась, а устройство за ним ещё числится, инженер должен узнать об этом сразу.

Каталог глазами гостя

[ Инженерный разбор ]

Чем эта система была сложна

Снаружи — «форма заявки». Внутри — пять несвязанных сред, которые должны видеть одно и то же состояние склада в реальном времени.

[ 01 ]

Бэкенд без фреймворка

Сервер написан на голом http-модуле Node.js — без Express и роутеров. Пришлось руками разбирать методы и пути, парсить тело запросов, отдавать статику Vite-сборки и держать REST API в одном процессе.

РешениеСобственный минималистичный роутинг-слой и вынос логики в api/handlers.js — предсказуемо, без «магии» и лишних зависимостей.
[ 02 ]

Остаток обязан сходиться

Заявка, выдача через бота, возврат, отмена, авто-возврат по концу сессии — все меняют один и тот же остаток. Гонки и двойные списания дают «минус одну мышь» или фантомный дефицит.

РешениеЕдиный источник правды по каждому устройству и атомарное изменение остатка на каждое событие жизненного цикла заявки.
[ 03 ]

Real-time без WebSocket-стека

Админ-панель должна показывать новые заявки и смену статусов мгновенно, но поднимать полноценный WS-сервер поверх голого Node было бы избыточно.

РешениеServer-Sent Events — однонаправленный поток событий держит админку актуальной без опроса и без тяжёлого WS-слоя.
[ 04 ]

Интеграция со SmartShell

Чтобы ловить «гость ушёл, а гарнитуру не сдал», сервис опрашивает биллинг SmartShell по GraphQL. Токен живёт недолго и его нужно молча обновлять, иначе поллинг падает среди ночи.

РешениеКлиент с авторизацией по login-мутации и автоматическим refresh-token; сессия закрылась без возврата — алерт инженеру.
[ 05 ]

Одна логика — три оболочки

Один и тот же фронтенд должен работать в браузере гостя, в режиме киоска и как десктоп-приложение Electron — без развилок и трёх разных кодовых баз.

РешениеЕдиная Vue 3 + Vite сборка, поверх которой тонкая Electron-обёртка — десктоп получает нативное окно, не дублируя UI.
[ 06 ]

Деплой без простоя

Сервис крутится на реальном сервере клуба и обновляется в рабочие дни. Ручной деплой на таком проекте — это регрессии в прод по пятницам.

РешениеКонтейнеризация в Docker и пайплайн GitHub Actions; выкладка скриптом deploy.sh, состояние переживает пересборку образа.

[ Стек и подходы ]

На чём собрано

Фронтенд
Vue 3 + Vite. Каталог, корзина, админ-панель. Реактивное состояние склада, компонентная витрина устройств.
Бэкенд
Чистый Node.js без фреймворков. Собственный HTTP-роутинг, REST API, отдача статики, JSON-хранилище.
Десктоп
Electron. Тонкая обёртка над той же Vue-сборкой — нативное окно без дублирования UI.
Бот
Telegram Bot API. Выдача/возврат устройств инлайн-кнопками, уведомления о заявках, алерты.
Интеграция
SmartShell GraphQL. Мониторинг игровых сессий, авторизация + refresh-token, алерты при завершении сессии без возврата.
Real-time
Server-Sent Events. Живые обновления заявок и статистики в админке без опроса.
Инфраструктура
Docker + GitHub Actions. Контейнеризация, CI/CD, деплой на живой сервер клуба скриптом.
